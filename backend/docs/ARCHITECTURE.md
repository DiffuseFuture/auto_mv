## 1. 应用分层架构图

系统按业务职责自顶向下划分为 7 层：

```
用户接入层      →  Web / 移动端 / 小程序 + 个人作品库 + 第三方开放接口 + 运营管理后台
业务编排层      →  智能对话编排 + 开放能力编排 + 会话与任务管理
AI 创作助手层   →  创作助手主控 + 意图路由 + 脚本生成专家 + 字幕转写专家 + 对话记忆 + SOP 剧本库
能力工具层      →  状态广播 + MV 创作工具集 + 音乐创作工具集
业务服务层      →  作品生成与合成服务 + 商业化服务 + 用户与社区服务
多模型适配层    →  图像生成 / 视频生成 / 音乐语音 / 口型同步 / 音乐结构分析
基础设施层      →  媒体处理引擎 + 对象存储 + 关系数据库 + 分布式缓存 + 安全与可观测
```

**关键业务设计**：

- **"双轨入口"**：
  - **C 端对话路径** —— 用户在 MV 创作工作台里与 AI 创作助手对话，实时看到主体图、分镜、视频片段一步步生成。
  - **B 端开放接口路径** —— 第三方 ISV / 平台型客户通过 APIKey 调用标准化能力（歌词 / 音乐 / MV），按异步任务处理。
- **AI 创作助手"主控 + 专家"分工**：主控负责与用户对话、按业务剧本（SOP 模板）把任务拆解到"专家" —— 脚本生成专家负责 MV 整体规划，字幕转写专家负责字幕时间线，对话记忆负责多轮上下文。
- **能力工具层按业务意图路由**：当用户要做 MV → 路由到 MV 创作工具集；要做音乐 → 路由到音乐创作工具集。
- **多模型适配层是"模型矩阵"**：图像 / 视频 / 音乐 / 口型 / 节拍分析都接入了多个模型供应商，平台按风格、画幅、成本智能调度。

## 2. "三强"约束映射图

围绕 MV 创作"**强音乐结构、强时间轴约束、强主体一致性**"三大核心业务诉求，平台在多个能力上做了专门设计。

### 2.1 强音乐结构 🎵 —— 让作品"听得懂节奏"

> **业务诉求**：用户希望 MV 不是"画面跟音乐凑在一起"，而是"画面在节拍上跳舞"，副歌、间奏、转场都精准匹配。

| 承担能力 | 业务价值 | 触发时机 |
| --- | --- | --- |
| 歌词创作能力 | 自然语言需求 → 标准化歌词（风格 / 标题 / 正文三段式） | 音乐剧本第 3 步 |
| 歌曲生成能力 | 支持人声 / 纯音乐双模式，男声 / 女声 / 自定义风格 | 音乐剧本第 4 步 |
| 多模型负载均衡 | 国际音乐生成模型 + 国内音乐生成模型双供应商，失败自动切换 | 上述生成时 |
| 音乐结构智能分析 | 自动识别 BPM、节拍点、段落（主歌 / 副歌 / 间奏） | MV 全局规划时 |
| 分镜节奏对齐 | 把节拍数据作为上下文喂给脚本生成 AI，输出与节奏对齐的分镜 | MV 全局规划时 |
| 音频淡入淡出 / 合成 | 作品末尾 2 秒自然淡出，音轨与画面精准合并 | 音视频合成时 |

**对外承诺**：
- 单个分镜时长不超过 10 秒
- 切镜时间精度达到 0.001 秒
- 同一主体在所有分镜中保持一致
- 同一首歌曲的时间戳可被后续 MV 脚本生成直接复用

### 2.2 强时间轴约束 ⏱ —— 让作品"对得上节拍"

> **业务诉求**：用户希望歌词字幕出现在正确的字、人声特写对上正确的拍、视频总长度等于歌曲长度，不会"差几秒尾巴"。

| 承担能力 | 业务价值 | 触发时机 |
| --- | --- | --- |
| 脚本时间精度 | 每个分镜拥有精确的开始 / 结束 / 时长（毫秒级） | MV 全局规划 |
| 字幕时间线对齐 | 字幕行时间戳与分镜精准对齐 | MV 全局规划 |
| 唱歌识别能力 | 针对歌唱场景的语音识别，输出 LRC / SRT 级别时间戳 | MV 全局规划 |
| 视频生成缓存 | 5 小时智能缓存，避免用户调整时分镜被重复生成 | 视频片段生成 |
| 媒体精确裁剪与适配 | 把模型输出精裁到目标时长；720p / 1080p 横竖屏自适应 | 视频片段生成 |
| 最终音视频合成 | 分镜拼接 + 音轨合并 + 末段淡出，总时长与音乐完全一致 | 作品合成 |
| 口型同步能力 | 按分镜时长精确裁剪音频片段，嘴型与歌声逐帧对齐；失败自动回退 | 作品合成 |
| 字幕烧录 | 支持中英文 + 自定义字体，按画幅自动调整字号 | 作品合成 |

**贯穿全程的时间数据**：MV 总时长、分镜数、单分镜开始 / 结束 / 时长、字幕行时间戳 + 文本 —— 这些数据在"脚本生成 → 字幕识别 → 视频裁剪 → 合成淡出"中逐跳传递，保证整条链路上的毫秒级一致。

### 2.3 强主体一致性 👤 —— 让作品"认得出主角"

> **业务诉求**：用户希望主角"小美"在第 1 镜、第 5 镜、第 12 镜里长得是同一个人，不会每跳一次就换脸、换衣服、换发型。

| 承担能力 | 业务价值 | 触发时机 |
| --- | --- | --- |
| 统一主体档案 | 每个作品维护一份"主体池"（1-7 个），含身份 / 类型 / 描述 / 风格参考 | MV 脚本生成 |
| 主体参考图生成 | 批量并行生成（最多 10 路并发），每次生成都形成独立版本 | MV 主体图生成 |
| 参考图复用机制 | 同一主体在不同分镜中复用同一份参考图，跨镜头视觉一致 | MV 主体图生成 |
| 主体池集中管理 | 主体结果图 ID 被永久锁定，不被意外覆盖 | MV 主体图生成 |
| 分镜主体绑定 | 将分镜中"主体 ID"自动解析为"实际参考图 ID"，作为视频模型统一输入 | 视频片段生成 |
| 主体 / 分镜在线编辑 | 新增 / 修改 / 删除三态支持；修改时自动检测并触发对应分镜重生成 | MV 全流程 |
| 作品库内二次编辑 | 在已发布作品上单独重做某个主体或某个分镜，保留原版本供切换 | 资源库 |
| 用户参考图理解 | 用户上传的参考图自动理解，并提示与预期是否匹配，避免误用 | MV 创意理解 |

**贯穿全程的主体数据**：主体档案（身份 / 类型 / 描述 / 风格）→ 主体参考图 ID → 分镜主体引用表（主体 ID + 实际文件 ID）→ 主体图历史版本（可切换）。

**特别注意**：主体图与视频生成必须在同一模型账号下生成，平台在多模型智能调度时已默认遵循此约束。

## 3. MV 创作端到端流程

每次 MV 创作严格遵循 8 步标准作业流程（SOP），每一步都在用户可控下推进，**用户未确认绝不进入下一步**，避免一通操作后"返工难"。

### 流程骨架

```
[创作者]  →  对话编排 (实时流式会话)  →  AI 创作助手 (加载 MV 创作剧本)
   │
   ├─ Step 1   创意理解 — 提取 音频 / 风格 / 画幅 / 歌词 / 参考图
   ├─ Step 1.5 参考图智能理解 — 自动描述 + 匹配度评估 (匹配度低时主动提醒)
   ├─ Step 2   完整性校验 (核心红线) — 缺音频则终止生成, 引导上传 / 现创作
   ├─ Step 3   任务初始化 — 为本次创作分配唯一任务编号, 贯穿后续所有步骤
   ├─ Step 4   全局规划 ★ 并行执行
   │           ├─ 音乐结构分析 (BPM / 节拍 / 段落)              🎵⏱
   │           ├─ 风格标准化
   │           ├─ 唱歌识别 (字幕时间线)                          ⏱
   │           └─ 脚本生成 → 输出 分镜表 + 主体表                🎵⏱
   ├─ Step 5   主体参考图生成 👤
   │           ├─ 多主体并发生成 (最多 10 路)
   │           ├─ 调用图像模型, 锁定参考图
   │           ├─ 历史版本可回溯, 切换
   │           ⚠ 等待用户确认主体图才能继续
   ├─ Step 6   分镜脚本确认 👤⏱
   │           ├─ 将主体融入分镜
   │           ├─ 推送到前端分镜展示
   │           ⚠ 等待用户确认分镜才能继续
   ├─ Step 7   视频片段生成 👤⏱
   │           ├─ 主体参考 ID 绑定, 传入视频模型
   │           ├─ 多分镜并发生成 (最多 10 路)
   │           ├─ 视频模型智能调度
   │           ├─ 媒体精裁到目标时长 + 分辨率适配
   │           ├─ 5 小时智能缓存避免重复生成
   │           ⚠ 等待用户确认视频片段才能继续
   └─ Step 8   音视频合成 ⏱🎵
               ├─ 可选: 按分镜对口型 (嘴型同步)                 ⏱
               ├─ 分镜画面拼接
               ├─ 合并歌曲 + 末段 2 秒淡出                       🎵
               ├─ 免费用户自动加水印 + 品牌片尾
               └─ 可选: 字幕烧录到画面
                       →  →  →  作品入库  →  实时推送给用户
```

### "三强"在 8 步中的分布

| 步 | 主要约束 | 失败时回退 |
| --- | --- | --- |
| Step 4 全局规划 | ⏱ + 🎵 | 监控告警；进度实时推送给用户 |
| Step 5 主体图生成 | 👤 | 单图重试；推送错误提示 |
| Step 7 视频生成 | 👤 + ⏱ | 5h 缓存兜底；推送错误提示 |
| Step 8 作品合成 | 🎵 + ⏱ | 口型同步失败时回退到原视频 |

### 端到端体验亮点

- **每一步都"看得见"**：用户在工作台能看到主体图、分镜、视频片段一步步生成，无需"等一个黑盒"返回。
- **每一步都"拦得住"**：用户对任一步不满意可即时修改，前序成果（已确认的主体图、已确认的分镜）会被复用，不会从头再来。
- **每一步都"算得清"**：积分按"作品类型 + 分辨率 + 视频模型"自动计算，提交时一次性校验，不足直接拒绝，避免"做了一半被卡"。

## 4. 音乐创作支线

系统对外提供两种音乐制作入口，对应不同客户类型。

### 4.1 路径 A：智能对话（面向 C 端用户）

- 用户在 MV 创作工作台里说"帮我写一首关于夏天的歌"
- AI 创作助手识别意图 → 加载音乐创作剧本
- 先出歌词 → 用户可改 → 再出歌曲
- 歌曲生成采用"国际音乐模型 + 国内音乐模型"双供应商，主备自动切换
- 末尾水印自动去除
- 积分自动扣减，作品入库
- 实时推送给用户

**音乐创作剧本（4 步标准作业流程）**：
1. 提取 风格 / 是否纯音乐 / 歌词 / 标题
2. 风格必填校验
3. 分支 A / B / C（纯音乐 / 用户歌词 / 自动歌词）
4. 调用歌曲生成

### 4.2 路径 B：开放接口（面向 ISV / 平台型客户）

- 第三方开发者通过 APIKey 调用标准接口
- 异步执行，状态可查询
- 提供"歌词生成 / 音乐生成 / MV 一键生成"三类标准能力
- 提供能力描述文档（歌词 / 音乐 / MV / 上传），降低接入成本
- 提交时一次性积分校验，不足直接拒绝

**MV 一键生成**：从脚本规划到音视频合成，全链路一次提交，全程自动化，与对话路径使用完全相同的底层能力。

### 4.3 两条路径的"三强"对比

| 业务约束 | 智能对话路径 | 开放接口路径 |
| --- | --- | --- |
| 🎵 强音乐结构 | 歌词创作 + 歌曲生成，多维度可控（性别 / 纯音乐 / 歌词） | 与对话路径共用同一组音乐生成模型 |
| ⏱ 强时间轴 | 词曲时间戳可被后续 MV 脚本生成直接引用 | MV 全链路与对话路径完全一致的时间精度 |
| 👤 强主体一致性 | 音乐支线无主体图，不涉及 | MV 全链路内主体参考图与视频生成一致 |

**结论**：两条路径在"音乐结构 / 时间轴 / 主体"上**共用底层能力**，差异只在"用户交互形式"（实时对话 vs 标准接口）。

## 5. 核心数据模型关系图

### 5.1 数据族谱

```
MV 脚本 (AI 生成)
├── 主体列表 (1..7 个)  ──→  主体档案 (ID / 类型 / 描述 / 风格参考)
└── 分镜表 (1..N 个)    ──→  分镜 (开始 / 结束 / 时长 / 主体引用 / 画面描述 / 是否口型同步)

统一任务载荷 (贯穿 MV 全流程的单一可信源)
├── 主体列表  ──→  主体档案 + 锁定主体图 ID  👤
├── 分镜表    ──→  分镜 + 主体引用解析表 (主体 ID → 实际文件 ID)  👤
│                + 口型同步文件 ID  ⏱
│                + 智能缓存键 (5 小时)
├── 音频文件 ID
└── 字幕时间线 (字幕行: 开始 / 结束 / 文本)  ⏱

音乐结构数据 (BPM / 节拍 / 段落)  →  注入脚本生成的 AI 上下文

主体引用解析结果
└── 主体 ID + 主体图 ID  ←  视频模型直接消费的格式
```

### 5.2 实时推送消息格式

- **初始化 / 文字 / 思考过程**
- **主体图（含历史版本）** 👤
- **分镜（含历史版本）** 👤⏱
- **分镜脚本 / 视频 / 音频 / 歌词**
- **错误 / 完成**

**历史回看**：用户在工作台查看历史作品时，系统自动从数据库恢复消息流，并把文件 ID 重新解析为可访问的预览地址。

### 5.3 持久化与传输

- **关系数据库**：
  - 用户消息（输入）
  - 流式消息片段（AI 输出）
  - 任务表（载荷 JSON）
  - 作品表（成品）
- **分布式缓存**：
  - 实时消息队列（对话进度推送）
  - 任务结果缓存（5 小时避免重复生成）
  - 会话锁（防止同一会话并发冲突）
  - 分享链接缓存（3 天有效）
  - 口型同步临时文件（24 小时）

## 7. 与技术方案的"三强"对应关系总览

| 技术方案表述 | 本架构图对应 | 关键实现 |
| --- | --- | --- |
| 强音乐节奏 | 2. 三强映射 🎵 / 3. MV 流程 Step 4 | 音乐结构分析把节拍注入脚本生成；歌曲生成带词曲时间戳 |
